Part 1

Briefly explain de Casteljau's algorithm and how you implemented it in order to evaluate Bezier curves.

The de Casteljau algorithm evaluates the bezier curve from a list of control points and parameter t. This algorithm has list.size() - 1 steps, and each step is described as follows: we take the linear interpolation (defined as t * points[i] + (1 - t) * points[i + 1]) between adjacent points in the list (but not between the first control point and the last control point) and the list of control points in the next step consists of all of these linearly interpolated control points. At the very end, there will be one item in the list, and that point will be the point on the curve at time step t. The curve consists of the points generated by taking points from all possible time steps t (from 0 to 1, inclusive).

Part 2

Briefly explain how de Casteljau algorithm extends to Bezier surfaces and how you implemented it in order to evaluate Bezier surfaces.

The de Casteljau algorithm extends to Bezier surfaces as well. If you have a 2D matrix (of dim N by N) of points and want to generate a Bezier surface from it, you first want to split the points into N groups of N vertices, and for each group of points, you generate N bezier curves with the parameter named u to draw the entire curve. Then, from each of these Bezier curves, you pick a point using the parameter u and take the list of points you get from these curves to generate another Bezier curve. Now you have bezier curves in two different dimensions, and you can use the parameters u and v to generate points to plot out the entire bezier surface.

In my implementation, I first wrote the evaluate1D function to generate one point a 1D bezier curve with parameter t. This function calls the evaluateStep function that I created in Part 1 and reassigns the parameter vector to this result of evaluateStep. I repeatedly call the evaluateStep function while the vector’s size > 1 to generate one final point. Then, to evaluate the entire bezier surface, I create a vector called intermediate_vector to hold the points generated with the evaluate1D function. I call the evaluate1D function with parameter u on each row of controlPoints, and finally, I return the result of calling evaluate1D on intermediate_points and parameter v and get one point back. Thus, when passing in different values of u and v to the evaluate function, I get back the entire bezier surface.

Part 3

Briefly explain how you implemented the area-weighted vertex normals.

I implemented the area-weighted vertex normals by first getting a halfedge named iterating halfedge by calling halfedge(). Then, I obtained the vertex’s position that belonged to that halfedge by calling halfedge()->vertex->position and named that vertex position A. Then, for each face, I obtained the two other vertices B and C that made up this triangle. I advanced the iterating halfedge by assigning it to its next() value, and then taking the vertex from this halfedge. I named this vertex B. Then, I advanced the iterating halfedge again by assigning it to its next() value, and then taking the vertex from this halfedge. I named the vertex C. Upon obtaining vertices B and C, I created vectors ab = b - a and ac = c - a. Then, I calculated the area of the face by using the formula area = 1/2 * ||ab|| * ||ac|| * sin(theta), where theta calculated by taking the inverse cosine of the result obtained from this formula: cos(theta) = (ab ᐧ ac)/(||ab|| * ||ac||). Then, I advanced the iterating halfedge by assigning it to its twin() value. I repeated this loop until the iterating halfedge equaled halfedge().

Part 6

Briefly explain how you implemented the loop subdivision and describe any interesting implementation / debugging tricks you have used.

We implemented loop subdivision by first computing the new positions of the vertices after loop subdivision occurs, and we store these new positions in the vertex's instance variable called newPosition. There are two different types of vertices.

First are the vertices that are not created during loop subdivision. For these vertices, we loop through all existing vertices, and for each vertex, we set the isNew field to be false and sum up the coordinates of all neighboring vertices. Then, we set a value u to be 3/16 if the number of neighbors is 3; otherwise we set u to be 3/(8 * the number of neighbors). Then, the new vertex position is (1 - number of neighbors * u) * the vertex's position + u * the sum of the positions of all neighboring vertices.

Then are the vertices that will be created during loop subdivision. Since we split an edge, we will loop through all edges. For each edge, we first set the isNew property to false and then obtain the four neighboring vertices: a, b, c, and d, where b and c are the two vertices on the edge and a and d are the vertices closest to b and c that form a triangle abd and bcd. Then, we set the edge's newPosition instance variable to be (3/8) * (a's position + b's position) + (1/8) * (c's position + d's position).

Then, we loop through all existing edges and split each edge, making sure to mark the vertex returned from the splitEdge function call's isNew property to true.

The fourth step is to flip edges that connect old and new vertices. We only want to split new edges, so we modify our split edge function to set the isNew property on the new edges that are created during the split and copy the edge's newPostion onto the newly created vertex's newPosition. Now, we iterate through all edges, and if the edge is new and one vertex on the edge is new and the other vertex is not new, we flip the edge.

The fifth step is to set the new positions of the vertices. We loop through all vertices, and if the vertex is new, we set its position to be its newPosition property.


Take some notes, as well as some screenshots, of your observations on how meshes behave after loop subdivision. What happens to sharp corners and edges? Can you reduce this effect by pre-splitting some edges?

We see that after each subdivision, edges and vertices become less pointy and sharp, and they become more curved, rounded out, and less pronounced. This is because the new position of the vertices is a a weighted average of the position of the vertices around it, and since a vertex along a sharp corner or edge is far more extreme in position than its neighboring vertices, the new position is in a less extreme position, therefore reducing the sharpness of the corner or edge. We can reduce this effect by splitting the edges adjacent to sharp corners and edges so that each neighboring pixel is more pronounced than before. These new adjacent vertices are in a more extreme position than the pre-split adjacent vertices.
